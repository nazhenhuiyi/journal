# Git 同步冲突处理需求

这份文档记录 Git 同步遇到真实内容冲突、无共同祖先历史断裂时的产品需求和实现边界。目标不是让同步永远成功，而是在无法安全自动判断时，不静默丢失用户日记内容。

## 背景

日记同步现在基于 GitHub 私有仓库和 `@journal/sync` 共享核心。正常情况下，双端改动可以通过 Git fetch / merge / push 串起来；Markdown 日记优先使用文本 diff3 合并。

问题出现在两类场景：

- 两端同时修改同一篇 Markdown 的同一段或相邻段落，文本合并无法 clean merge。
- 本地仓库和远端仓库没有共同祖先，说明同步历史断裂，可能来自换仓库、重建仓库、误清 Git 目录或测试数据污染。

这两类场景都不能用“自动 last-write-wins”解决。对日记应用来说，同步失败但两边内容还在，比偷偷覆盖其中一边更可接受。

## 产品原则

- 不静默丢内容：应用不能在用户不知道的情况下替用户丢掉本机或远端内容。
- 不自动推送冲突标记：如果产生 conflict markers，只能作为本地待处理草稿或诊断状态，不能自动 commit / push 到远端。
- 同步核心先保证安全：冲突时可以失败，但失败状态必须是应用能理解的结构化状态。
- 用户操作必须显式：保留本机、保留远端、合并草稿、一次性修复历史，都必须由用户主动确认。
- 普通网络、认证、权限错误仍按普通同步错误处理；内容冲突和历史断裂是“需要处理”的阻塞状态，不应该密集重试。

## 术语

- 内容冲突：同一受管文件两边都有改动，自动文本合并无法判断应保留哪一边。
- 无共同祖先：本地分支和远端分支找不到 merge base。
- 阻塞状态：同步不能继续自动 push，需要用户进入处理流程。
- 本机版本：当前设备 worktree 里的版本。
- 远端版本：GitHub 远端分支上的版本。
- 合并草稿：带冲突标记或人工拼接内容的本地草稿，必须由用户编辑确认后才能再次同步。

## P0：安全阻断

第一步先补安全闭环，不要求立刻做复杂 diff UI。

### 同步核心

- Markdown true conflict 必须停止自动 push。
- 无共同祖先不能默认自动合并并推送。
- 同步核心需要抛出结构化阻塞错误，至少区分：
  - `content-conflict`
  - `unrelated-histories`
- 结构化错误需要保留用户可读 message，并尽量带上诊断信息：
  - conflict path count
  - 本地 commit oid
  - 远端 commit oid
  - merge base 结果
  - branch / remote
- 如果底层 Git merge 已经把 conflict markers 写入本地 worktree，同步核心仍然不能 commit / push；后续同步也要识别当前仓库处于待处理状态。

### 同步调度

- 新增或等价表达一个 `blocked` 同步状态。
- `blocked` 不自动进入 retry timer。
- `blocked` 不应被后台 pull 静默覆盖成“已同步”。
- 本地 pending changed paths 不能因为进入 `blocked` 被清空。
- 只有用户完成处理、修改同步配置、或显式执行修复流程后，才能离开 `blocked`。

### 移动端状态展示

- 首页轻状态可以显示“需要处理”。
- 设置页同步区域需要显示明确原因：
  - 内容冲突：`有内容冲突，需要处理后再同步。`
  - 历史断裂：`本机和远端不是同一条同步历史，需要一次性修复。`
- 手动“立即同步”遇到阻塞时，alert 标题应区别于普通失败，例如：
  - `需要处理同步冲突`
  - `需要修复同步历史`
- 调试日志保留原始错误和 trace，但 UI 不展示长堆栈。

## P1：简化冲突处理入口

第一版处理入口不需要完整 diff 编辑器，但必须给用户清楚的选择。

### 入口

- 设置页在 `blocked` 状态下显示“处理同步问题”入口。
- 如果能定位到具体日期或路径，入口文案展示对应日期；否则展示文件数量。
- 处理页进入前不触发新的自动 push。

### 内容冲突处理

对单个 Markdown 冲突，提供三种选择：

1. 保留本机版本
   - 使用本机文件内容作为最终版本。
   - 完成后创建解决冲突的同步提交，再 push。
   - push 前需要重新 fetch，确认远端没有新的不同改动；如果又变化，重新进入阻塞状态。

2. 保留远端版本
   - 使用远端文件内容覆盖本机文件。
   - 今日页面和相关缓存需要重新加载。
   - 如果本机还有其他未冲突改动，仍保留 pending 同步。

3. 合并为草稿
   - 生成本地草稿，包含本机和远端两边内容，必要时带 conflict markers。
   - 用户必须打开草稿编辑并保存。
   - 保存后再次同步前，应用需要检测该文件不再包含未处理 conflict markers。

首版可以只展示本机/远端两段纯文本和三个按钮，不要求行级 diff。更理想的后续版本再做左右对照、高亮差异和逐段选择。

## P1：无共同祖先修复

无共同祖先不是普通内容冲突。它说明本机和远端历史关系不可信，不能自动把“更长的一边”当祖先。

### 为什么不能默认把长的一边作为祖先

在 Git 三方合并里，base 的含义是“共同过去”。如果把更长的一边硬当 base，另一边看起来就像删除了很多文件或内容，合并结果可能反而丢掉更完整的一边。

“更长的一边”只能作为修复流程里的推荐主历史，不能作为自动合并规则。

### 修复入口

当检测到 `unrelated-histories`：

- 显示本机和远端摘要：
  - 受管文件数量
  - 最近提交时间
  - 最近提交信息
  - 本机独有文件数量
  - 远端独有文件数量
  - 同名但内容不同文件数量
- 默认推荐内容更多或最近使用的一边作为主历史，但必须让用户确认。
- 用户可以选择：
  - 以远端为主，补回本机独有内容。
  - 以本机为主，补回远端独有内容。
  - 暂不修复，只保留本机写作。

### 修复规则

- 独有文件可以复制到主历史。
- 同名且内容相同的文件直接保留。
- 同名但内容不同的 Markdown 进入内容冲突处理。
- 非 Markdown 同名不同内容不能自动覆盖，进入需要确认状态。
- 默认不做破坏性 force push。
- 如果未来必须提供 force push，需要单独危险确认，并建议先创建远端备份分支。

## P2：更完整的处理体验

后续可以增强：

- 冲突列表，按日期排序。
- 左右对照 diff。
- 逐段选择本机或远端。
- 自动保存合并草稿。
- 处理完成后展示“将推送 1 个解决冲突提交”的确认摘要。
- 桌面端和移动端共享同一套冲突状态模型。

## 非目标

- 不把 GitHub REST API 作为默认同步协议。
- 不恢复系统 `git` 命令行 fallback。
- 不在后台自动 last-write-wins 解决 Markdown true conflict。
- 不自动 force push 覆盖远端。
- 不要求首版做复杂 diff 编辑器。

## 验收标准

P0 完成时：

- 双端同时修改同一 Markdown 冲突段落后，应用不 push。
- 远端仓库不会出现 conflict markers。
- 同步状态进入“需要处理”或等价阻塞状态。
- 设置页能展示内容冲突原因。
- 保存后的 pending changes 不会被错误清空。
- 单元测试覆盖 commit / push 没有被调用。

P1 内容冲突入口完成时：

- 用户可以选择保留本机版本并成功同步。
- 用户可以选择保留远端版本并刷新本机页面。
- 用户可以生成合并草稿，编辑后再次同步。
- 未处理 conflict markers 的草稿不能被自动 push。

P1 历史修复完成时：

- 无共同祖先不会自动合并并推送。
- 用户能看到本机和远端摘要。
- 用户明确选择主历史后，独有内容被补回。
- 同名不同内容进入冲突处理，而不是被静默覆盖。
