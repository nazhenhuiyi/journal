# Journal Sync 目标与边界

这份文档从用户故事和产品需求出发，说明且留为什么需要 `journal sync`，它承诺解决哪些问题，哪些问题必须明确阻断或交给用户选择。实现细节见 [Git 同步机制](<Git 同步机制.md>)、[Git 同步冲突处理](<Git 同步冲突处理.md>) 和 [E2E 测试](<E2E 测试.md>)。

## 1. 产品目标

且留的同步不是“给应用加一个 Git 功能”，而是让一个人可以在不同设备上继续同一份日记生活。

用户真正需要的是：

- 在桌面端写长日记，晚上拿起手机还能看到同一份内容。
- 在手机上随手记碎碎念、加图片，回到桌面端不会丢。
- 当前设备离线、切后台、网络差、GitHub 暂时失败时，写作本身不被打断。
- 多端都改过内容时，能自动合并可以安全合并的部分；不能安全合并时，不要假装成功。
- 同步状态可以被理解：正在同步、已同步、需要登录、稍后重试、需要人工处理。
- 用户的数据仍然是一份普通 GitHub 私有仓库，而不是被锁进某个自建后端。

所以 `journal sync` 的核心目标是：

```txt
本地优先写作 + 私有 Git 仓库归档 + 多端最终一致 + 遇到不确定时停止伤害
```

从用户角度看，“同步成功”不是按钮变绿，而是同时满足：

- 刚写下的内容仍然在本机可读。
- 远端私有仓库包含应该同步的日记数据。
- 本地 worktree 和当前分支处在可继续写作的状态。
- 没有未解决冲突被偷偷提交或推送。
- 如果另一端有新内容，当前端已经在安全时机读盘刷新，或者明确标记为待处理。

## 2. 用户故事与承诺

| 用户故事 | 同步承诺 |
| --- | --- |
| 新设备填写 GitHub 仓库和 token | token 只进平台凭据存储；仓库地址不能携带凭据；空本地可以从远端初始化；本地和远端都有独立历史时必须让用户选择方向 |
| 日常写作后自动同步 | 先保证本地保存；用 `changedPaths` 限制检查范围；连续输入 debounce；输入或保存不稳定时延后；push 被远端更新拒绝后 fetch / merge / retry 一次 |
| 打开应用或回到前台 | 静默检查远端；远端没变就不做无意义 fetch / merge；确认 tracking commit 可读；本地还有未提交内容时不覆盖；远端落盘后由应用安全刷新 |
| 手机离线、弱网或切后台 | 离线和网络失败保留 pending；后台 flush 有时间上限；重试克制，避免密集 push 或反复下载大包；网络恢复后继续 |
| 多端同时修改同一天 | Markdown 先走 diff3，再尝试日记结构合并；`updatedAt` 不决定整篇 Markdown 胜负；无法判断的同段冲突必须停止自动 push |
| 图片和结构化文件 | 图片作为整体文件同步；review / annotation JSON 优先按可靠时间字段选较新；没有可靠时间时按当前方向默认选边 |
| 远端仓库里有非日记文件 | 应用本地提交只提交日记受管范围；merge 遇到非日记路径时远端胜出；应用不为这些路径提供编辑或语义合并 |
| 本地 Git 对象库异常 | 目标 commit 必须实际可读；缺 `.idx` 时优先本地重建 pack index；修复失败再清 remote tracking ref 并 refetch；同一缺失对象的 refetch 要节流 |

## 3. 同步数据边界

且留承诺同步的是用户日记数据和少量跨端派生数据：

```txt
entries/
media/
annotations/
reviews/
manifest.json
```

这些路径的含义：

| 路径 | 用户意义 | 同步策略 |
| --- | --- | --- |
| `entries/` | 长日记、碎碎念、天气和位置 front matter | Markdown 日记合并，冲突时停止 |
| `media/` | 用户导入的图片和媒体 | 文件整体同步，冲突时远端版本胜出 |
| `annotations/` | 旧批注数据 | JSON 时间字段选较新，否则选边 |
| `reviews/` | 当天回顾展示快照 | JSON 时间字段选较新，否则选边 |
| `manifest.json` | 仓库级元数据预留 | 结构化文件策略 |

不属于同步真相：

- 当前页面状态、滚动位置、输入法 composing 状态。
- 移动端 widget snapshot 本地缓存。
- 桌面端本地索引。
- token、用户设置 UI 草稿、诊断日志。
- 测试 runner 的临时分支、临时目录和 E2E 注入配置。

## 4. 系统职责边界

### 4.1 `journal sync` 应该负责

- 管理本地 Git 仓库初始化、clone、commit、fetch、domain merge、push。
- 只提交受管日记范围内的本地改动。
- 用完整 refs 维护分支，避免 detached HEAD 和短 ref。
- 判断远端是否变化，并验证本地 tracking 对象可读。
- 执行日记领域合并，生成 fast-forward 或 merge commit。
- 对可恢复 Git 对象库问题做本地自愈。
- 给应用层返回 dirty paths、sync result、错误和 trace。
- 维护跨端一致的调度状态机：pending、syncing、synced、retrying、needs-auth、error。
- 记录足够定位问题的同步 trace，但不记录 token、Authorization header、请求体、日记正文或完整私密 URL。

### 4.2 应用层应该负责

- 决定什么时候可以同步：是否正在输入、保存、切后台、页面能否刷新。
- 保存日记内容并产出可信 `changedPaths`。
- 管理 token 和同步配置的 UI。
- 把同步状态展示成用户能理解的文案。
- 在远端更新落盘后，安全地重新读盘并刷新 UI、review、widget snapshot。
- 提供冲突处理入口、首次同步方向选择等需要用户决策的体验。
- 管理诊断包导出、日志脱敏和用户可读的修复建议。

### 4.3 用户决策边界

只要结果取决于用户意图，而不是仓库事实，`journal sync` 就不应该在后台自动决定。具体需要阻断的场景见下一节。

## 5. 自动处理边界

这些情况系统应该尽量自动处理：

| 场景 | 自动策略 | 用户可见结果 |
| --- | --- | --- |
| 连续保存 | debounce 后合并成一次 push | 少量提交，状态最终已同步 |
| 当前仍在输入 | 延后 push / pull | 保持 pending，不打断写作 |
| 远端无变化 | 跳过 fetch / merge | 状态安静，不产生提交 |
| push 被远端更新拒绝 | fetch -> domain merge -> retry push once | 可以合并时自动完成 |
| 网络失败 | 进入 retrying，稍后重试 | 不丢本地内容 |
| tracking commit 不可读 | 先补 pack index，再必要时 refetch | 尽量省流量自愈 |
| 旧 ref 布局问题 | 修复 HEAD / 短 ref | 不让仓库停在游离状态 |
| 远端更新后 UI 需要刷新 | 应用层安全读盘 | 不覆盖正在编辑的内存状态 |

自动处理的前提是“结果确定且可验证”。只要无法确定不会丢内容，就应该停下来。

## 6. 必须阻断边界

这些情况不能假装同步成功：

| 场景 | 为什么阻断 | 应给用户的方向 |
| --- | --- | --- |
| Markdown true conflict | 系统无法判断哪段文字代表用户意图 | 进入冲突处理入口 |
| 本地已有 conflict markers | 继续提交会把冲突标记推到远端 | 先解决冲突再同步 |
| 首次同步本地和远端都有历史 | 自动提交会制造历史断裂 | 让用户选择方向 |
| 本地和远端无共同祖先 | 合并语义不可信 | 做一次性迁移或重建 |
| token 缺失、失效或损坏 | 无法认证远端 | 重新登录 / 保存 token |
| remote URL 内含凭据 | 有泄露风险 | 拒绝保存地址 |
| 对象库修复后仍不可读 | 无法证明 merge 输入完整 | 节流后提示修复 |
| 受管范围外的本地编辑 | 应用不理解语义 | 不作为日记内容提交 |

阻断不是失败体验的终点。产品上还需要把这些状态从底层错误升级为明确的“需要处理”入口。

阻断状态至少应该能区分：

- 需要登录或重新保存 token。
- 有内容冲突，需要用户处理。
- 本地和远端历史不兼容，需要选择迁移方向。
- 本地同步仓库损坏，需要修复或重新拉取。
- 当前正在写作或保存，稍后自动继续。

## 7. 非目标

`journal sync` 不应该变成：

- 通用 Git 客户端。
- 团队协作文档系统。
- GitHub REST 文件同步器。
- 任意仓库文件编辑器。
- 大媒体资产管理和历史压缩工具。
- 备份恢复 UI 的全部实现。
- 冲突可视化编辑器本身。

这些能力可以围绕同步继续建设，但不应该塞进 Git core 的主路径里。

## 8. 需求驱动的架构

从用户故事反推，当前同步架构应该保持四层：

```txt
用户写作体验
  -> 平台 sync manager：判断何时安全同步、如何提示和刷新 UI
  -> JournalSyncCoordinator：单飞、debounce、pending paths、retry
  -> Git core：仓库事实、受管路径提交、journal-domain merge、push/fetch
  -> 平台 runtime：fs / http / credentials / trace
```

这条分层的重点不是“代码好看”，而是保护产品承诺：

- 写作体验可以本地优先，不被 Git 操作拖住。
- 调度状态可预测，不会并发跑多个 Git 操作。
- Git core 可以在桌面和手机复用同一套合并规则。
- 平台差异被限制在文件系统、HTTP、凭据和生命周期。
- 测试可以在不同层级模拟真实风险，而不是只测 UI 文案。

## 9. 拟真验收故事

同步的验收应该围绕用户故事，而不是只看某个函数返回值。

| 用户故事 | 可信验收 |
| --- | --- |
| 桌面写日记，手机读到 | 真实 GitHub 测试仓库；桌面 push 后移动端或隔离 clone 能读到 `entries/` 内容 |
| 手机写碎碎念，桌面读到 | 移动端原生 E2E 写入并手动同步；远端分支存在对应 Markdown |
| 多端非重叠修改同一天 | 两个临时 worktree 制造共同祖先和分叉提交；domain merge 后两边内容都在 |
| 两端同段冲突 | merge 停止，保留本地待处理，不 push 冲突结果 |
| 图片冲突 | 二进制文件整体选边，worktree 和提交树一致 |
| 远端无变化 | 不 fetch、不 merge、不产生新 commit |
| push 期间远端先更新 | 第一次 push 被拒绝后 fetch / merge / retry 成功 |
| 弱网或认证失败 | 状态进入 retrying 或 needs-auth，本地内容仍在 |
| 手机对象库缺 `.idx` | 构造 orphan pack；优先 indexPack 修复；成功时不重新下载大包 |
| 首次同步双边都有内容 | 停止并提示需要选择方向 |
| detached HEAD / 短 ref 回归 | 同步后 HEAD 附着在 `refs/heads/<branch>`，无 `.git/main` |
| 诊断日志导出 | 能定位 sync 阶段和错误类型，但不包含 token、headers、正文或完整私密 URL |

测试分层建议：

- Unit：日记 merge、JSON 选边、状态展示、调度状态机。
- Integration：真实 `isomorphic-git` 临时仓库、对象库修复、Expo 文件系统适配、SecureStore/pending paths。
- App E2E：真实 Electron 或原生移动端，把“保存 -> 同步 -> 远端存在 -> UI 状态”串起来。
- GitHub E2E：只用专用私有测试仓库和临时分支；不要使用真实日记仓库。

## 10. 当前体验缺口

这些是从用户故事看仍然欠缺的产品能力：

- 冲突还缺专门页面：用户不能直观看到本机 / 远端 / 合并草稿。
- 阻断状态还不够结构化：`error` 里混有认证、冲突、历史断裂、对象库异常等不同问题。
- 首次同步方向选择还需要 UI：保留本地、导入远端、合并为草稿。
- 大媒体治理还不足：同步能搬运图片，但不能帮助用户理解仓库体积。
- 对象库修复失败后的用户指引还偏底层：应该解释为“本地同步仓库损坏，需要修复或重新拉取”，而不是暴露 Git 错误。

这些缺口不改变当前同步核心的边界，但会决定下一阶段产品体验要补在哪里。

## 11. 决策原则

以后改 `journal sync` 前，先按这几条判断：

1. 这能不能保护用户刚写下的内容？
2. 这会不会把不确定的合并结果推到远端？
3. 这是不是日记应用需要理解的文件类型？
4. 这是不是应该由用户明确选择，而不是后台自动决定？
5. 这有没有让手机端在后台反复下载、提交或重试？
6. 这能不能用真实多端故事验收，而不是只靠 mock？

答案不清楚时，默认停下来、保留本地内容、让状态可解释。
