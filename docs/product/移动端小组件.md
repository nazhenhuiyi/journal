# 移动端小组件

这份文档记录移动端小组件当前实现和边界。早期 Android 选型调研已经落地，本文只保留仍需要维护的事实。

## 1. 当前状态

移动端已经接入两套原生小组件能力：

| 平台 | 实现 | 入口 |
| --- | --- | --- |
| Android | `react-native-android-widget` | `apps/mobile/src/widgets/JournalMomentAndroidWidget.tsx` |
| iOS | `expo-widgets` + `@expo/ui/swift-ui` | `apps/mobile/src/widgets/JournalMomentWidget.ios.tsx` |

两端共享同一种 `JournalWidgetSnapshot` 数据，不共享渲染实现。小组件只读本地 snapshot，不直接跑 Git 同步，也不直接扫描日记仓库。

## 2. Snapshot 数据流

```txt
日记保存 / Review 加载 / 远端更新 / Widget 更新请求
  -> journalEffects
  -> refreshJournalWidgetSnapshot()
  -> listDailyJournals()
  -> loadOrCreateDailyReview()
  -> createJournalWidgetSnapshot()
  -> 写入 journal-widget-snapshot-v1.json
  -> updateNativeJournalWidgets()
```

关键文件：

- `apps/mobile/src/services/journalWidgetSnapshotStore.ts`
- `apps/mobile/src/services/journalEffects.ts`
- `packages/journal-core/src/journalWidgetSnapshot.ts`
- `packages/journal-core/src/reviewMoments.ts`

`journal-widget-snapshot-v1.json` 是移动端本地缓存，不进入 Git 同步范围。真正需要跨端同步的是 `entries/` 和按需生成的 `reviews/YYYY/MM/YYYY-MM-DD.json`。

## 3. Snapshot 模式

`JournalWidgetSnapshot` 当前有两类展示：

- `theme-entry`：低摩擦写作入口，例如“记一件小事”。
- `review-moment`：从旧日记中挑出一条回顾内容。

点击行为通过 deep link 表达：

```txt
journal://write?theme=<themeId>
journal://review-day?date=<YYYY-MM-DD>
journal://review
```

小组件渲染层只负责展示 title、subtitle、accent color 和点击链接，不重新计算推荐逻辑。

## 4. Android

Android 使用 `react-native-android-widget`：

- `apps/mobile/index.ts` 在 Android 平台注册 `registerWidgetTaskHandler()`。
- `widgetTaskHandler.tsx` 响应 `WIDGET_ADDED`、`WIDGET_UPDATE`、`WIDGET_RESIZED`。
- `JournalMomentAndroidWidget.tsx` 使用 widget 专用组件渲染 RemoteViews。
- `journalWidgetNative.android.tsx` 调用 `requestWidgetUpdate()` 主动刷新已添加的小组件。

`apps/mobile/app.json` 中的 Android widget 配置名称是 `JournalMoment`，展示文案是“且留”。

边界：

- Android widget UI 不是普通 React Native View，不能直接复用应用页面组件。
- 后台更新频率受 Android 系统和厂商省电策略影响。
- 小米 / HyperOS 等系统应以“可添加到桌面、点击可打开、App 内保存后可刷新”为验收目标，不承诺分钟级准时后台刷新。

## 5. iOS

iOS 使用 Expo SDK 56 的 `expo-widgets`：

- `JournalMomentWidget.ios.tsx` 通过 `createWidget()` 定义 SwiftUI 风格 widget。
- `journalWidgetNative.ios.ts` 调用 `JournalMomentWidget.updateSnapshot(snapshot)`。
- `apps/mobile/app.json` 配置 `expo-widgets`，bundle id 是 `app.zilin.journal.widgets`。
- `withExpoWidgetsTargetVersion` config plugin 负责补齐 widget target 的版本和字体资源。

边界：

- iOS widget 使用 SwiftUI 风格组件，不能直接渲染普通 React Native 页面。
- Widget Extension 的资源和字体要显式进入 target。
- 需要 development build / EAS Build 验收。

## 6. 刷新触发

当前刷新入口：

- 日记保存成功。
- Review 页面加载并按需创建 review 文件。
- 同步应用了远端更新。
- Android 系统触发 widget update / resize / added。
- App 主动调用 `updateNativeJournalWidgets()`。

刷新失败时只记录错误，不阻断日记保存或同步。小组件可以继续显示上一次有效 snapshot 或 fallback 文案。

## 7. 验收清单

通用：

- 保存今天的日记后，snapshot 文件被更新。
- 远端 pull 更新本地日记后，snapshot 会刷新。
- 没有可展示内容时，小组件显示写作入口 fallback。
- 点击小组件能打开对应 deep link。

Android：

- 小组件出现在系统“添加小组件”列表。
- 添加到桌面后能展示当前 snapshot。
- 调整尺寸后布局不截断主要文字。
- 在小米 / HyperOS 真机上验证添加、点击、保存后刷新。

iOS：

- Development build 或 EAS build 包含 widget extension。
- Widget target 能加载字体资源。
- `JournalMomentWidget.updateSnapshot()` 后 widget 内容更新。
- `systemMedium` 布局不截断主要文字。

## 8. 不做什么

- 小组件不直接读取 GitHub。
- 小组件不直接执行同步。
- 小组件不做复杂编辑，只做入口和轻展示。
- 小组件不承诺高频后台刷新。
- 小组件 snapshot 不作为跨端权威数据。
