# 前端批注库调研

这份文档调研前端文本批注开源生态中两个相关项目：

- Recogito Text Annotator
- AnnotatorJS

目标是判断它们是否适合用于当前 Electron + React + Markdown 日记应用的 AI 批注系统。

## 当前产品方案背景

当前产品的批注方向是：

- Markdown 源文件是 truth。
- AI 批注不写入主日记文件。
- 批注旁路存储。
- 预览态通过 Markdown AST 给块级节点挂 `data-annotation-ids`。
- 编辑态批注暂缓。

因此，一个外部库是否适合，核心取决于它是否能尊重 Markdown 源码和 AST 坐标系，而不是只围绕渲染后的 DOM 文本流工作。

## 结论

不建议直接接入 AnnotatorJS。

Recogito Text Annotator 可以局部借鉴，或在很小范围内试点，但不建议把它作为核心 truth 或锚定层。

当前自研 `Markdown-aware annotation anchoring` 仍然是更稳的方向。

## Recogito Text Annotator

### 项目地址

- GitHub: [recogito/text-annotator-js](https://github.com/recogito/text-annotator-js)
- npm: [@recogito/text-annotator](https://www.npmjs.com/package/%40recogito%2Ftext-annotator)

### 项目现状

Recogito Text Annotator 是一个用于 Web 应用的文本批注库。

项目 README 中写明它提供：

- Text Annotator。
- React wrapper。
- TEI/XML extension。
- Recogito PDF Annotator。

当前生态相对现代：

- TypeScript 为主。
- 有 Vite 相关开发结构。
- 有 React wrapper。
- License 是 BSD-3-Clause。

npm registry 中 `@recogito/text-annotator` 当前版本为 `3.4.9`，React wrapper `@recogito/react-text-annotator` 也存在同版本。

GitHub 规模不大，但仍比 AnnotatorJS 更现代。

### 主要能力

核心入口：

```ts
import { createTextAnnotator } from '@recogito/text-annotator'

const anno = createTextAnnotator(document.getElementById('content'))
```

主要能力包括：

- 加载 annotation。
- 创建 annotation。
- 更新 annotation。
- 删除 annotation。
- undo / redo。
- annotation filter。
- annotation style。
- 控制 visible / selected。
- scrollIntoView。
- 用户信息。
- viewport intersection。

主要事件包括：

- `createAnnotation`
- `updateAnnotation`
- `deleteAnnotation`
- `selectionChanged`
- `viewportIntersect`

React wrapper 提供：

- `TextAnnotator`
- `TextAnnotationPopup`
- `useAnnotator`
- `useSelection`
- `useViewportState`

### 数据模型

Recogito 的数据模型接近 W3C Web Annotation，但不是完全原生 W3C。

README 中示例结构大致是：

```json
{
  "id": "67a427d7-afc3-474a-bdab-1e2ea8dc78f6",
  "bodies": [],
  "target": {
    "selector": [
      {
        "quote": "Tell me, O muse",
        "start": 48,
        "end": 63
      }
    ],
    "creator": {
      "id": "zD62eVrpvJgMEEWuPpPS"
    },
    "created": "2025-09-30T07:28:54.973Z",
    "updated": "2025-09-30T07:28:56.158Z"
  }
}
```

特点：

- `bodies` 承载评论、标签或应用自定义 payload。
- `target.selector` 承载选中文本 quote 和字符 offset。
- 结构接近 W3C 的 `body + target + selector` 思路。

源码中还有 W3C adapter，可以序列化为包含：

- `@context: http://www.w3.org/ns/anno.jsonld`
- `type: Annotation`
- `TextQuoteSelector`
- `TextPositionSelector`

### 高亮与锚定机制

Recogito 强依赖 DOM Range。

大致流程：

1. 用户在预览 DOM 中选择文本。
2. 从浏览器 selection / Range 计算 `quote`、`start`、`end`。
3. 恢复时遍历 DOM text nodes。
4. 根据 offset 重新构造 `document.createRange()`。
5. 渲染时通过 CSS Custom Highlight API、`Range.getClientRects()` overlay、span 或 canvas 等方式画高亮。

源码层面可以看到它支持三种 renderer：

```ts
type RendererType = 'SPANS' | 'CANVAS' | 'CSS_HIGHLIGHTS'
```

默认 renderer 是 `SPANS`。

如果配置 `CSS_HIGHLIGHTS`，它会先检测 `CSS.highlights` 是否存在；如果不支持，则回退到默认 `SPANS`。

### Recogito 渲染链路

Recogito 的渲染链路大致是：

```txt
annotation selector
-> revive 成 DOM Range
-> Range 转 client rects
-> rects 进入 spatial tree
-> renderer 根据可见区域绘制
```

#### selector 恢复成 Range

Recogito 保存的 selector 主要包含：

```ts
quote
start
end
```

恢复时，它遍历 container 下所有 text node，根据 `start/end` 找到开始和结束 text node，然后用 `document.createRange()` 重建 DOM Range。

这说明它的锚点是 DOM 可见文本流 offset，不是 Markdown source offset。

相关源码：

- [`reviveSelector.ts`](https://raw.githubusercontent.com/recogito/text-annotator-js/main/packages/text-annotator/src/utils/reviveSelector.ts)

#### Range 转 rects

Recogito 不只是简单调用一次 `range.getClientRects()`。

它有一个 `getHighlightClientRects(range)` 工具：

- 如果选区只在一个 text node 中，直接使用 `range.getClientRects()`。
- 如果选区跨多个 text node，则遍历相交 text node。
- 对第一个 text node 和最后一个 text node 创建局部 range。
- 对中间 text node 分别创建 range。
- 最后合并 rects。

这样能更稳地处理跨 `<strong>`、跨 inline node、跨多行文本的情况。

相关源码：

- [`getHighlightClientRects.ts`](https://raw.githubusercontent.com/recogito/text-annotator-js/main/packages/text-annotator/src/utils/getHighlightClientRects.ts)

#### spatial tree

Recogito 会把 annotation 的 rects 放入空间索引。

空间索引用于：

- 鼠标 hover 命中检测。
- 查找当前视口内有哪些 annotation。
- 快速找到 annotation 的边界。
- scrollIntoView。

它使用 `rbush` 管理这些矩形。

这个设计很值得借鉴：高亮不是只画出来，还需要参与交互和视口状态计算。

相关源码：

- [`spatialTree.ts`](https://raw.githubusercontent.com/recogito/text-annotator-js/main/packages/text-annotator/src/state/spatialTree.ts)

### Recogito 三种 renderer

#### SPANS renderer

这是默认 renderer。

它会在 container 中插入一个高亮层：

```html
<div class="r6o-span-highlight-layer"></div>
```

然后对每个 rect 创建一个绝对定位的 span：

```html
<span
  class="r6o-annotation"
  data-annotation="..."
  style="left: ...; top: ...; width: ...; height: ..."
></span>
```

注意：这里的 `span` 不是包裹原文文字，而是 overlay rect。它比直接修改文本节点要干净，但仍然是 DOM overlay 层。

相关源码：

- [`spansRenderer.ts`](https://raw.githubusercontent.com/recogito/text-annotator-js/main/packages/text-annotator/src/highlight/span/spansRenderer.ts)

#### CSS_HIGHLIGHTS renderer

它会动态写入 `::highlight(...)` 样式，并把 Range 注册到 `CSS.highlights`：

```ts
const highlight = new Highlight(...ranges)
CSS.highlights.set(`_${annotation.id}`, highlight)
```

这种方式不修改 DOM，浏览器负责基础文字高亮。

注意：源码中会调用 `CSS.highlights.clear()`。如果我们自己实现，不应该全局清空所有 highlights，而应该只管理自己的 key，避免影响页面里其它高亮。

相关源码：

- [`highlightsRenderer.ts`](https://raw.githubusercontent.com/recogito/text-annotator-js/main/packages/text-annotator/src/highlight/highlights/highlightsRenderer.ts)

#### CANVAS renderer

它通过 canvas 根据 rects 画高亮。

源码中有 warning：Canvas renderer 已 deprecated，并建议使用默认 `SPANS` renderer。

因此 canvas 路线只适合参考，不适合直接采用。

相关源码：

- [`canvasRenderer.ts`](https://raw.githubusercontent.com/recogito/text-annotator-js/main/packages/text-annotator/src/highlight/canvas/canvasRenderer.ts)

其中比较值得我们借鉴的是分层思路：

```txt
CSS Custom Highlight API：负责文字底色
Range.getClientRects() overlay：负责交互和精致视觉层
```

CSS Highlight 不修改 DOM，适合做基础文字高亮。  
`getClientRects()` overlay 样式更自由，适合 hover、选中态、侧边连接线、气泡锚点和动画。

这两者可以结合使用，但它们仍然属于预览态渲染层，不应该成为批注持久化的 truth。

更准确地说，Recogito 从 DOM Range 开始，而我们应该从 Markdown source range 开始，最后才生成 DOM Range：

```txt
Recogito:
DOM Range
-> rects
-> spatial tree
-> renderer

本产品:
Markdown source range
-> Markdown AST / source map
-> DOM Range
-> CSS Highlight 基础底色
-> getClientRects overlay 交互增强
```

相关源码包括：

- [`rangeToSelector.ts`](https://raw.githubusercontent.com/recogito/text-annotator-js/main/packages/text-annotator/src/utils/rangeToSelector.ts)
- [`reviveSelector.ts`](https://raw.githubusercontent.com/recogito/text-annotator-js/main/packages/text-annotator/src/utils/reviveSelector.ts)
- [`getHighlightClientRects.ts`](https://raw.githubusercontent.com/recogito/text-annotator-js/main/packages/text-annotator/src/utils/getHighlightClientRects.ts)

### 与当前方案的适配性

适合的场景：

- 预览 DOM 内用户手动选中文本。
- 需要快速获得一个可用的文本高亮交互层。
- 批注不需要与 Markdown 源码长期稳定绑定。

不适合的地方：

- 它的 `start/end` 是相对渲染 DOM 文本流的 offset。
- Markdown 源码和预览 DOM 文本不是同一个坐标系。
- Markdown 插件、重新渲染、隐藏元素、代码块、链接、脚注、HTML entity 等都可能改变 DOM 文本 offset。
- 它不能天然解决 Markdown AST block position 到批注的映射。

因此，Recogito 不适合作为核心锚定层。

### 对当前产品的建议

建议：局部借鉴，谨慎试点，不作为核心依赖。

可借鉴：

- W3C-ish annotation shape。
- `bodies` / `purpose` 的扩展方式。
- `TextQuoteSelector + TextPositionSelector` 双 selector。
- CSS Highlight + overlay 的渲染分层。
- hover / selected / style API。
- viewport intersection。
- React wrapper 的交互组织方式。

如果未来试点：

- 只把它作为预览态临时交互层。
- 不让它决定持久化 truth。
- 持久化仍使用我们自己的 Markdown source range、AST block range 和 AI annotation id。
- 编辑态继续暂缓。

## AnnotatorJS

### 项目地址

- GitHub: [openannotation/annotator](https://github.com/openannotation/annotator)
- Docs: [docs.annotatorjs.org](https://docs.annotatorjs.org/en/latest/)
- npm: [annotator](https://www.npmjs.com/package/annotator)

### 项目现状

AnnotatorJS 是非常经典的网页批注库，历史影响力大。

GitHub 数据显示：

- 约 2.7k stars。
- 约 538 forks。
- 最新 GitHub release 是 `Annotator v1.2.10`，发布时间为 2015-02-26。

npm 中 `annotator` 最新版本是 `2.0.0-alpha.3`。

官方 latest docs 对 2.0 文档有明确警告：这是 bleeding-edge documentation，可能不准确、不稳定。稳定文档应看 1.2.x。

技术栈偏旧：

- 1.x 文档要求 jQuery 1.6+。
- 2.x 是 Browserify / Webpack 时代模块。
- 插件体系偏老。
- License 情况需要谨慎，GitHub 显示 MIT / GPL / Unknown 文件并存，npm license 为空。

### 主要能力

AnnotatorJS 提供：

- 网页文本选择。
- annotation UI。
- editor。
- viewer。
- filter。
- tags。
- permissions。
- auth。
- store plugin。

1.x 初始化方式：

```js
$('#content').annotator()
```

插件方式：

```js
$('#content').annotator('addPlugin', 'Store', {
  prefix: '/store'
})
```

Store plugin 会根据 create / update / destroy / search 等事件向 REST JSON API 同步。

### 数据模型

AnnotatorJS 的数据模型不接近现代 W3C Web Annotation。

它的 annotation 大致包含：

```json
{
  "id": "...",
  "annotator_schema_version": "...",
  "text": "...",
  "quote": "...",
  "uri": "...",
  "ranges": [
    {
      "start": "...",
      "end": "...",
      "startOffset": 0,
      "endOffset": 10
    }
  ],
  "tags": [],
  "permissions": {}
}
```

其中 `ranges.start/end` 是 XPath 相关位置。

这不是 `body + target + selector` 的现代 W3C 结构。

### 高亮与锚定机制

AnnotatorJS 依赖 DOM selection / range。

它通过 `xpath-range` 把范围序列化和反序列化。

高亮方式是把文本节点包入：

```html
<span class="annotator-hl">...</span>
```

这意味着它会实际修改 DOM。

相关源码：

- [`textselector.js`](https://raw.githubusercontent.com/openannotation/annotator/main/src/ui/textselector.js)
- [`highlighter.js`](https://raw.githubusercontent.com/openannotation/annotator/main/src/ui/highlighter.js)

### 与当前方案的适配性

不适配。

主要原因：

- 它锚定的是渲染后的 DOM XPath + offset。
- 它会包裹 DOM 节点。
- Markdown AST 重新生成 DOM 后，XPath 很容易漂移。
- jQuery/plugin/store 体系不适合当前 Electron + React 状态管理。
- 数据模型不贴近我们当前设计的 `body + target + selector`。

### 对当前产品的建议

建议：不用作依赖，只作为历史参考。

可以参考：

- annotation UI、store、permissions、tags 拆成插件的思路。

不要采用：

- XPath range 模型。
- DOM span 注入式 highlighter。
- jQuery 插件架构。
- 旧 store plugin 体系。

## 综合建议

### 不推荐直接使用的原因

这两个库都更偏向“在已经渲染好的网页 DOM 中做文本批注”。

当前产品需要的是：

```txt
Markdown source
-> longEntryMarkdown source offset
-> Markdown AST block position
-> data-annotation-ids
-> 预览态轻标记
```

也就是说，核心问题不是“如何在 DOM 上画 highlight”，而是“如何让批注稳定地附着在 Markdown 源文件表达的内容上”。

### 推荐路线

继续自研 `Markdown-aware annotation anchoring`。

具体策略：

- 存储层采用自己的 `AnnotationFile`。
- selector 借鉴 W3C 和 Recogito 的 `TextQuoteSelector + TextPositionSelector`。
- 预览态由 Markdown AST 负责块级挂载。
- UI 交互可以参考 Recogito。
- 不直接使用 AnnotatorJS。
- 不把 DOM Range 作为长期 truth。

一句话结论：

> Recogito 可参考，AnnotatorJS 可考古；核心锚定层仍应自己实现。
