# 数据结构

这份文档只记录目前已经讨论清楚的数据结构方向，后续继续迭代。

## 核心原则

每天的日记保存为一个 Markdown 文件：

```txt
entries/YYYY/MM/YYYY-MM-DD.md
```

例如：

```txt
entries/2026/04/2026-04-24.md
```

一天就是一个完整容器。长日记、碎碎念、图片等内容都属于这一天。

Markdown 文件是主要数据源。未来如果需要数据库，数据库更适合作为索引和缓存，而不是唯一真相。

## 文件结构

建议目录：

```txt
journal/
  manifest.json
  entries/
    2026/
      04/
        2026-04-24.md
        2026-04-25.md
  media/
    2026/
      04/
        window-rain.jpg
        cactus.jpg
  annotations/
    2026/
      04/
        2026-04-24.json
```

## Front Matter

每个日记文件顶部使用 Front Matter 保存这一天的基础元数据。

```md
---
date: 2026-04-24
createdAt: 2026-04-24T21:10:00+08:00
updatedAt: 2026-04-24T23:20:00+08:00
weather:
  text: 小雨
  temperature: 18
location:
  name: 上海
---
```

Front Matter 只放这一天的背景信息，不放对这一天的意义判断。

适合放：

- 日期。
- 创建时间。
- 更新时间。
- 天气。
- 地点。

不适合放：

- 情绪评分。
- 心理判断。
- 对用户状态的推断。

## 长日记

Front Matter 之后的普通 Markdown 正文就是当天长日记。

```md
# 今天

今天的正文从这里开始。
```

长日记是一日一篇，偏连续表达。

碎碎念 block 统一放在长日记正文之后的文末区域，不插入长日记正文中。

## 碎碎念

碎碎念使用自定义 block 表达。

碎碎念不是纯文本，而是一天里的轻量记录。它可以包含文字，也可以包含图片。

```md
:::murmur
id: m_20260424_213800
time: 2026-04-24T21:38:00+08:00
---
窗外下雨了，声音很轻。
:::
```

字段：

- `id`：碎碎念唯一标识。
- `time`：碎碎念发生或创建时间。

## 图片

图片可以出现在碎碎念中。图片不是日记附件，而是某个瞬间记录的一部分。

图片不允许独立出现在当天文件中，必须属于某条碎碎念。

图片使用 `::image` 子 block。每张图片可以有自己的 `caption`、`tags` 和可选 `location`。

```md
:::murmur
id: m_20260424_213800
time: 2026-04-24T21:38:00+08:00
---
窗外下雨了，声音很轻。

::image
id: img_20260424_213801
src: ../media/2026/04/window-rain.jpg
caption: 雨打在窗户上
tags: [雨, 窗户, 夜晚]
location:
  name: 上海市徐汇区
  latitude: 31.1885
  longitude: 121.4365
  source: exif
::
:::
```

图片字段：

- `id`：图片记录唯一标识。
- `src`：图片文件路径。
- `caption`：图片说明。
- `tags`：图片标签。
- `location`：可选位置数据。

标签目前只放在图片上，长日记和碎碎念暂时不设置标签。

位置字段：

- `name`：可选，地点名称。
- `latitude`：可选，纬度。
- `longitude`：可选，经度。
- `source`：可选，位置来源。

位置来源可以是：

- `exif`：从图片 EXIF 信息提取。
- `manual`：用户手动填写。
- `system`：由系统或应用环境提供。

如果图片没有位置信息，则不写 `location`。

```md
::image
id: img_20260424_213801
src: ../media/2026/04/window-rain.jpg
caption: 雨打在窗户上
tags: [雨, 窗户, 夜晚]
::
```

## 只有图片的碎碎念

碎碎念可以没有文字，只有图片。

```md
:::murmur
id: m_20260424_215000
time: 2026-04-24T21:50:00+08:00
---

::image
id: img_20260424_215001
src: ../media/2026/04/desk.jpg
caption: 晚上的桌面
tags: [桌面, 灯光, 夜晚]
::
:::
```

这表示：即使用户写不出文字，也可以用一张图片留下今天。

## 多图片碎碎念

一条碎碎念可以包含多张图片，每张图片都有独立元数据。

```md
:::murmur
id: m_20260424_214500
time: 2026-04-24T21:45:00+08:00
---
今天看了一眼仙人掌。

::image
id: img_20260424_214501
src: ../media/2026/04/cactus-1.jpg
caption: 还是没有开花
tags: [仙人掌, 窗边]
::

::image
id: img_20260424_214502
src: ../media/2026/04/cactus-2.jpg
caption: 靠近看有一点新的刺
tags: [仙人掌, 细节]
::
:::
```

## 当前类型草案

```ts
type DayFrontMatter = {
  date: string
  createdAt: string
  updatedAt: string
  weather?: {
    text?: string
    temperature?: number
  }
  location?: {
    name?: string
  }
}

type MurmurBlock = {
  id: string
  time: string
  body: string
  images: ImageBlock[]
}

type ImageBlock = {
  id: string
  src: string
  caption?: string
  tags: string[]
  location?: ImageLocation
}

type ImageLocation = {
  name?: string
  latitude?: number
  longitude?: number
  source?: 'exif' | 'manual' | 'system'
}
```

## 批注兼容

批注界面已经移除，但旧数据结构继续保留，避免未来读写旧日记时丢失旁路数据。

原则：

- 新功能不主动创建批注。
- 移动端第一版不显示批注。
- 读写日记 Markdown 时不能删除已有批注文件。
- 批注文件仍然是旁路 JSON，不写入 `entries/YYYY/MM/YYYY-MM-DD.md`。
