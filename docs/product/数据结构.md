# 数据结构

这份文档记录当前已经落地的数据结构约定。代码事实以 `packages/journal-core` 为准；修改格式时需要同时考虑桌面端、移动端和同步仓库里的历史数据。

## 核心原则

每天的日记保存为一个 Markdown 文件：

```txt
entries/YYYY/MM/YYYY-MM-DD.md
```

例如：

```txt
entries/2026/04/2026-04-24.md
```

一天就是一个完整容器。长日记、碎碎念、图片等内容都属于这一天。

Markdown 文件是主要数据源。未来如果需要数据库，数据库更适合作为索引和缓存，而不是唯一真相。

## 文件结构

当前目录：

```txt
journal/
  manifest.json
  entries/
    2026/
      04/
        2026-04-24.md
        2026-04-25.md
  media/
    2026/
      04/
        window-rain.jpg
        cactus.jpg
  annotations/
    2026/
      04/
        2026-04-24.json
  reviews/
    2026/
      06/
        2026-06-10.json
```

## Front Matter

每个日记文件顶部使用 Front Matter 保存这一天的基础元数据。桌面端和移动端保存时都会保留未知字段，并维护 `date`、`createdAt` 和 `updatedAt`。

```md
---
date: 2026-04-24
createdAt: 2026-04-24T21:10:00+08:00
updatedAt: 2026-04-24T23:20:00+08:00
weather:
  text: 小雨
  temperature: 18
  feelsLike: 17
  humidity: 82
  windSpeed: 11
  updatedAt: 2026-04-24T21:20:00+08:00
location:
  name: 上海
  region: 上海
  country: 中国
---
```

Front Matter 只放这一天的背景信息，不放对这一天的意义判断。

适合放：

- 日期。
- 创建时间。
- 更新时间。
- 天气。
- 地点。

不适合放：

- 情绪评分。
- 心理判断。
- 对用户状态的推断。

## 长日记

Front Matter 之后的普通 Markdown 正文就是当天长日记。

```md
# 今天

今天的正文从这里开始。
```

长日记是一日一篇，偏连续表达。

碎碎念 block 统一放在长日记正文之后的文末区域，不插入长日记正文中。

## 碎碎念

碎碎念使用自定义 block 表达。

碎碎念不是纯文本，而是一天里的轻量记录。它可以包含文字，也可以包含图片。

```md
:::murmur
id: m_20260424_213800
time: 2026-04-24T21:38:00+08:00
themes: [small-thing]
---
窗外下雨了，声音很轻。
:::
```

字段：

- `id`：碎碎念唯一标识。
- `time`：碎碎念发生或创建时间。
- `themes`：可选，稳定 theme id 数组；前台显示中文名，旧记录没有该字段时按空数组处理。

`themes` 是记录入口和回顾锚点，不是前台标签管理。今日页不需要主动展示 theme 选择器；当用户从小组件、回顾或低干扰入口创建碎碎念时，应用可以把对应 theme 写入这里。

## 图片

图片可以出现在碎碎念中。图片不是日记附件，而是某个瞬间记录的一部分。

图片不允许独立出现在当天文件中，必须属于某条碎碎念。

图片使用 `::image` 子 block。每张图片可以有自己的 `caption`、`tags` 和可选位置字段。

```md
:::murmur
id: m_20260424_213800
time: 2026-04-24T21:38:00+08:00
---
窗外下雨了，声音很轻。

::image
id: img_20260424_213801
src: media/2026/04/window-rain.jpg
caption: 雨打在窗户上
tags: [雨, 窗户, 夜晚]
location: 上海市徐汇区
latitude: 31.1885
longitude: 121.4365
locationSource: exif
::
:::
```

图片字段：

- `id`：图片记录唯一标识。
- `src`：图片文件路径。
- `caption`：图片说明。
- `tags`：图片标签。
- `location`：可选地点名称。
- `latitude`：可选纬度。
- `longitude`：可选经度。
- `locationSource`：可选位置来源。

标签目前只放在图片上，长日记和碎碎念暂时不设置标签。

解析兼容：

- `location` 和 `locationName` 都会解析为 `ImageLocation.name`。
- `locationSource` 和 `location-source` 都会解析为 `ImageLocation.source`。
- `latitude` / `longitude` 只有在可解析为有限数字时才会进入结构化位置。

位置来源可以是：

- `exif`：从图片 EXIF 信息提取。
- `manual`：用户手动填写。
- `system`：由系统或应用环境提供。

如果图片没有位置信息，则不写 `location`。

```md
::image
id: img_20260424_213801
src: media/2026/04/window-rain.jpg
caption: 雨打在窗户上
tags: [雨, 窗户, 夜晚]
::
```

## 只有图片的碎碎念

碎碎念可以没有文字，只有图片。

```md
:::murmur
id: m_20260424_215000
time: 2026-04-24T21:50:00+08:00
---

::image
id: img_20260424_215001
src: media/2026/04/desk.jpg
caption: 晚上的桌面
tags: [桌面, 灯光, 夜晚]
::
:::
```

这表示：即使用户写不出文字，也可以用一张图片留下今天。

## 多图片碎碎念

一条碎碎念可以包含多张图片，每张图片都有独立元数据。

```md
:::murmur
id: m_20260424_214500
time: 2026-04-24T21:45:00+08:00
---
今天看了一眼仙人掌。

::image
id: img_20260424_214501
src: media/2026/04/cactus-1.jpg
caption: 还是没有开花
tags: [仙人掌, 窗边]
::

::image
id: img_20260424_214502
src: media/2026/04/cactus-2.jpg
caption: 靠近看有一点新的刺
tags: [仙人掌, 细节]
::
:::
```

## 回顾浮现

回顾浮现保存为稀疏 JSON 文件：

```txt
reviews/YYYY/MM/YYYY-MM-DD.json
```

只有当天真的生成出可展示的浮现内容时才写文件；没有内容时不创建空文件。`entries/` 仍然是用户内容事实，`reviews/` 只保存“这一天翻出了哪些旧纸条”的展示快照和来源日期。

```json
{
  "version": 1,
  "date": "2026-06-10",
  "generatedAt": "2026-06-10T20:00:00.000Z",
  "moments": [
    {
      "id": "anniversary-2025-06-10",
      "sourceDays": ["2025-06-10"],
      "themes": ["sky-now"],
      "anchors": [
        { "type": "date", "label": "那年今日", "value": "2025-06-10" },
        { "type": "theme", "label": "此刻的天空", "value": "sky-now" }
      ],
      "kind": "anniversary",
      "title": "那年今日",
      "subtitle": "你写过一句：风很好",
      "widgetEligible": true
    }
  ]
}
```

回顾文件可以跨端同步，但不复制完整日记正文或图片元数据。展示完整内容时，应用应根据 `sourceDays` 回到对应日文件读取。

## 当前核心类型

```ts
type DayFrontMatter = {
  date?: string
  createdAt?: string
  updatedAt?: string
  title?: string
  excerpt?: string
  tags?: string[]
  favorite?: boolean
  collections?: string[]
  weather?: {
    text?: string
    temperature?: number
    feelsLike?: number
    humidity?: number
    windSpeed?: number
    updatedAt?: string
  }
  location?: {
    name?: string
    region?: string
    country?: string
    query?: string
  }
  [key: string]: unknown
}

type MurmurBlock = {
  id: string
  time: string
  themes: string[]
  body: string
  images: ImageBlock[]
}

type ImageBlock = {
  id: string
  src: string
  caption?: string
  tags: string[]
  location?: ImageLocation
}

type ImageLocation = {
  name?: string
  latitude?: number
  longitude?: number
  source?: 'exif' | 'manual' | 'system'
}

type ReviewFile = {
  version: 1
  date: string
  generatedAt: string
  moments: ReviewMoment[]
}

type ReviewMoment = {
  id: string
  sourceDays: string[]
  themes: string[]
  anchors: ReviewAnchor[]
  kind: 'single' | 'cluster' | 'anniversary' | 'relative'
  title: string
  subtitle?: string
  widgetEligible: boolean
}

type ReviewAnchor = {
  type: 'date' | 'solarTerm' | 'weather' | 'timeOfDay' | 'season' | 'personal' | 'theme'
  label: string
  value?: string
}

type JournalWidgetSnapshot = {
  version: 1
  date: string
  generatedAt: string
  mode: 'theme-entry' | 'review-moment'
  title: string
  subtitle?: string
  footnote?: string
  action:
    | { type: 'write'; themeId: string }
    | { type: 'reviewDay'; date: string }
    | { type: 'review' }
}
```

完整类型定义见 `packages/journal-core/src/types.ts`。

## 批注兼容

批注界面已经移除，但旧数据结构继续保留，避免未来读写旧日记时丢失旁路数据。

原则：

- 新功能不主动创建批注。
- 移动端当前不显示批注。
- 读写日记 Markdown 时不能删除已有批注文件。
- 批注文件仍然是旁路 JSON，不写入 `entries/YYYY/MM/YYYY-MM-DD.md`。

## 容错与未定

- 损坏 Markdown 会返回 diagnostics，应用端负责提示用户；不会在解析阶段丢弃原文。
- 是否为日记列表建立可重建索引。
