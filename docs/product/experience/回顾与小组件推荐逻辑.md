# 回顾与小组件推荐逻辑

这份文档说明回顾内容怎么生成、怎么被选进小组件。端侧 widget 实现见 [移动端小组件](移动端小组件.md)。

## 数据角色

| 数据 | 作用 |
| --- | --- |
| `ReviewSourceDay` | 从日记文件读出的当天内容，包含长日记、murmur、图片和 frontmatter |
| `ReviewMoment` | 一个可回看的片段，说明哪一天因为什么线索值得被看见 |
| `ReviewFile` | 按天落盘的回顾文件，路径是 `reviews/YYYY/MM/YYYY-MM-DD.json` |
| `reviews/weekly/YYYY-Www.md` | 周回顾 Markdown，提供周回顾 title、summary 和可选封面 |
| `JournalWidgetBundleSnapshot` | 给原生小组件读取的本地缓存，包含 `review` 和 `moment` 两块 |

## Daily Review 怎么生成

入口是 `loadOrCreateDailyReview({ date, sourceDays })`：

1. 先读取当天 `ReviewFile`。
2. 如果文件合法，直接复用。
3. 如果不存在或不合法，调用 `createReviewMoments()` 生成 moments。
4. 有 moments 才写入 `ReviewFile`；没有内容就不写空文件。

`createReviewMoments()` 只看今天以前的日记，排除没有可回顾内容的日子。可回顾内容指长日记非空、murmur 有文字，或 murmur 有图片。

## Moment 类型

生成器按固定优先级补满最多 5 个 moment：

| 优先级 | 类型 | 生成条件 |
| --- | --- | --- |
| 1 | `anniversary` | 历史里有同月同日的日记 |
| 2 | `solarTerm` | 今天是节气，历史里有同一节气当天的日记 |
| 3 | `cluster` | 某个 theme 出现在至少 2 个不同日子 |
| 4 | `relative` | 历史里有上周或上个月的今天 |
| 5 | `single` | 最近历史日记补位 |

同一个 source day 只会进入一个 moment。越高优先级先占用日期，后续规则跳过已用日期。

## Moment 文案

title 规则：

| 类型 | title |
| --- | --- |
| `anniversary` | `那年今日` 加天气、节气或时间锚点 |
| `solarTerm` | `{节气名}那天` |
| `cluster` | `你留下过一些{主题短名}` |
| `relative` | `上周的今天` / `上个月的今天` 加锚点 |
| `single` | 日期加锚点，例如 `6 月 12 日，小雨` |

subtitle 来自当天代表性材料：

1. 合适短句：`你写过一句：...`
2. 图片 caption；没有 caption 时用 `你留过一张照片。`
3. theme：`{主题名}里的一小块。`

短句只从 murmur 里取，不从长日记正文里摘。长度需要在 6 到 28 个字符之间，并排除明显负向词，避免小组件把尖锐内容突然放到桌面。

## widgetEligible

`widgetEligible` 表示这个 moment 是否允许进入小组件推荐池。筛选条件：

- `widgetEligible === true`
- title 非空
- `sourceDays` 里至少有一个合法日期

目前正常生成出来的 moment 基本都会 eligible。这个字段主要留给未来：私密内容、敏感内容、只适合 Review 页但不适合桌面外露的内容。

## 回看怎么选内容

`createJournalWidgetBundleSnapshot()` 按顺序生成 `bundle.review`：

1. 新鲜周回顾：最新周回顾的 `endDate` 距今天不超过 1 天。
2. 每日回顾：从当天 `ReviewFile.moments` 里挑一个 moment。
3. 空占位：没有可展示回顾时，点击进入 `small-thing`。

周回顾要求 `week/startDate/endDate/title/summary` 合法。`coverImage` 是可选字段：合法的 `media/...` 路径会保留为 `backgroundImageSrc`，但当前小组件不展示封面；非法时只丢弃图片，不丢弃周回顾。

每日回顾选择顺序：

1. 先过滤 eligible moments。
2. 如果有强回顾，直接选第一个。强回顾包括 `anniversary`，或 anchors 里有 `solarTerm` / `weather`。
3. 否则从 eligible moments 里按 `hashString(date) % eligibleMoments.length` 稳定选一条。
4. 没有 eligible moment 时才展示 `empty-review`。

这里不再按历史积累做展示概率。「回看」负责尽量展示回顾；「此刻」负责表达入口。

## 此刻怎么选 theme

`bundle.moment` 固定是 `theme-entry`。它按本地时间段给候选 theme，并避开今天已经用过的同一 theme。

| 时间 | 候选 theme |
| --- | --- |
| 周末 09:00-17:00 | `breathe-moment`、`small-thing`、`quick-photo` |
| 05:00-10:00 | `sky-now`、`sunrise-sunset`、`small-thing` |
| 10:00-14:00 | `food-today`、`small-thing`、`today-three-lines` |
| 17:00-20:00 | `light-shadow`、`sunrise-sunset`、`sky-now` |
| 21:00-05:00 | `thought-maybe`、`small-thing`、`shower-thought` |
| 其他时间 | `small-thing`、`today-three-lines`、`quick-photo` |

如果候选都已经用过，就回退到候选列表里的第一个合法 theme。

## 刷新时机

`journalEffects` 统一协调刷新，避免保存、同步、进入前台和系统 widget 更新并发踩踏。

| 触发点 | 方法 |
| --- | --- |
| App 进入前台 | `refreshForAppActive` |
| 日记保存成功 | `afterJournalSaved` |
| Review 页面加载完成 | `afterReviewLoaded` |
| 同步应用远端更新 | `afterRemoteUpdatesApplied` |
| 日期跨天 | `afterDateRollover` |
| 原生小组件请求更新 | `refreshForWidgetUpdate` |

刷新会重新生成 `journal-widget-snapshot-v2.json`，并通知原生小组件更新。新建的 `ReviewFile` 会作为本地变更进入同步链路；widget snapshot 只是本地缓存，不同步。

## 稳定性边界

- moment 生成用固定优先级，不做模型打分。
- 小组件一次只展示一条回顾。
- 多条 eligible daily moment 的选择来自日期 hash，不来自运行时随机。
- 今天内容、远端同步、首次生成 ReviewFile 都可能改变当天 snapshot。
- 如果以后需要解释“为什么选中这一条”，优先给选择结果增加 `selectionReason`。
