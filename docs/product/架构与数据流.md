# 架构与数据流

这份文档记录当前仓库的模块边界和主数据流。历史规划、POC 记录和已完成 checklist 不再放在这里；需要验证实现时，以代码和各目录 `AGENTS.md` 为准。

## 1. 当前目标

且留是一个桌面端和移动端共用数据格式的个人日记应用。当前核心功能是：

- 写当天长日记。
- 记录碎碎念和图片。
- 回看已有日记。
- 生成回顾内容和小组件快照。
- 通过 GitHub 私有仓库在桌面端和移动端之间同步。

桌面端和移动端共享领域模型，不共享 UI。同步是 Git 同步，不是自建后端，也不是用 GitHub REST API 重新实现文件同步协议。

## 2. Workspace 分层

```txt
journal/
  apps/
    desktop/
      electron/        # Electron main、preload、Node 平台适配
      src/             # React renderer、桌面端页面和交互
    mobile/
      src/             # Expo / React Native 页面、服务和平台适配
      e2e/             # Maestro 原生 E2E flow
      plugins/         # Expo config plugins
      scripts/         # Expo / Maestro 启动脚本
      ios/             # 当前已生成的 iOS native 工程
  packages/
    journal-core/      # 日记格式、解析、序列化、review/widget/weather 领域纯函数
    journal-sync/      # Git 同步核心、调度状态机、合并策略、状态展示
    journal-theme/     # 跨端主题 token
  e2e/                 # Playwright 桌面端和真实 GitHub 同步 E2E
  docs/product/        # 产品与技术文档
```

根目录只做 pnpm workspace 编排。单个 workspace 命令优先使用：

```sh
pnpm --filter <workspace-name> run <script>
```

## 3. 共享包边界

| 包 | 职责 | 不做什么 |
| --- | --- | --- |
| `@journal/core` | 日记 Markdown、front matter、murmur、image、annotation、review、widget snapshot、weather 的类型和纯函数 | 不依赖 Electron、React Native、Node 文件系统、网络、UI |
| `@journal/sync` | Git 仓库状态、commit/fetch/merge/push、同步调度、dirty paths、冲突合并、状态展示 | 不保存 token，不读取平台设置，不直接依赖 Electron / Expo |
| `@journal/theme` | 颜色、间距、圆角等跨端设计 token | 不承载业务逻辑 |

应用端只做平台适配和交互编排：文件系统、凭据、安全存储、生命周期、页面状态、输入稳定性、真实窗口或真机验收。

## 4. 权威数据源

同步仓库中的文件是跨端权威数据源：

```txt
journal-worktree/
  manifest.json
  entries/
    2026/
      06/
        2026-06-10.md
  media/
    2026/
      06/
        img_20260610_220000.jpg
  annotations/
    2026/
      06/
        2026-06-10.json
  reviews/
    2026/
      06/
        2026-06-10.json
```

`entries/` 是用户内容事实；`media/` 保存图片文件；`annotations/` 保留旧批注 schema；`reviews/` 保存当天生成的回顾展示快照；`manifest.json` 留给仓库级元数据。索引、widget snapshot 和 UI state 都是可重建缓存或平台状态，不是跨端唯一真相。

## 5. 桌面端数据流

桌面端主链路：

```txt
CodeMirror / 碎碎念面板 / 图片导入
  -> renderer 组合长日记、murmurs、front matter
  -> @journal/core 序列化 Markdown
  -> preload IPC
  -> Electron main 写 entries/ 和 media/
  -> renderer 根据保存结果计算 changed paths
  -> desktopSyncManager.markLocalSave()
  -> JournalSyncCoordinator
  -> @journal/sync Git 操作
```

关键位置：

- `apps/desktop/src/pages/MarkdownPreviewPage.tsx`：编辑、自动保存、导入图片后汇总 tracked paths。
- `apps/desktop/electron/journalIndex.ts`：扫描 `entries/` 并生成桌面端本地索引。
- `apps/desktop/electron/journalMedia.ts`：导入图片到 `media/YYYY/MM/`。
- `apps/desktop/src/services/sync/desktopSyncManager.ts`：把保存状态、编辑器脏状态、手动同步和后台拉取接到共享调度器。
- `apps/desktop/electron/journalSync.ts`：Electron main 中的 Git runtime 和凭据适配。

桌面端保存后通常产生：

```txt
entries/YYYY/MM/YYYY-MM-DD.md
media/YYYY/MM/<image>
```

天气刷新会修改当天 `entries/...md` 的 front matter，也会进入同一条同步链路。

## 6. 移动端数据流

移动端主链路：

```txt
Today 页面输入 / 图片选择 / Review 页面
  -> mobileJournalStore 读写本地 worktree
  -> @journal/core 解析或序列化 Markdown / review JSON
  -> 保存结果返回 changedPaths
  -> journalEffects 处理保存后的副作用
  -> mobileSyncManager.markLocalSave()
  -> JournalSyncCoordinator
  -> mobileGitSync 创建 Expo runtime
  -> @journal/sync Git 操作
```

关键位置：

- `apps/mobile/src/services/mobileJournalStore.ts`：读写 `entries/`、`media/`、`reviews/`，保存后返回 changed paths。
- `apps/mobile/src/services/journalEffects.ts`：保存、review 加载、远端更新后的命名副作用协调。
- `apps/mobile/src/services/journalWidgetSnapshotStore.ts`：生成本地 widget snapshot，并通知原生小组件。
- `apps/mobile/src/services/sync/mobileSyncManager.ts`：输入稳定性、前后台、保存后延迟推送、手动同步和 pending paths。
- `apps/mobile/src/services/sync/mobileGitSync.ts`：Expo 文件系统、`expo/fetch`、SecureStore 凭据和 `@journal/sync` 的连接层。
- `apps/mobile/src/services/sync/pendingSyncPaths.ts`：移动端重启后恢复尚未推送的 tracked paths。

移动端保存后可能产生：

```txt
entries/YYYY/MM/YYYY-MM-DD.md
media/YYYY/MM/<image>
reviews/YYYY/MM/YYYY-MM-DD.json
```

`journal-widget-snapshot-v2.json` 是移动端本地给原生小组件读取的缓存，不在同步仓库范围内；缺少 v2 时移动端会读取旧 `journal-widget-snapshot-v1.json` 做兼容适配。

## 7. 同步数据流

保存后自动同步：

```txt
本地保存成功
  -> 返回 changedPaths
  -> 记录 pending changed paths
  -> 等待保存后 debounce
  -> 如果仍在输入或中文输入法 composing，则继续延后
  -> commit tracked changes
  -> push
```

打开 app / 回到前台 / 定时拉取：

```txt
生命周期触发
  -> 如果本地正在编辑，跳过这次后台 pull
  -> fetch 远端
  -> merge 到本地 worktree
  -> 如果 worktree 更新且当前页面可安全刷新，则重新读盘
  -> 刷新 review / widget snapshot 等派生内容
```

手动“立即同步”：

```txt
保存或补保存当前内容
  -> commit 本地 tracked changes
  -> fetch / merge 远端
  -> push
  -> 如果远端在 push 前又变化，fetch / merge 后重试一次 push
```

后台离开：

```txt
进入后台或页面解绑
  -> 尝试 flush 未保存正文
  -> 如果输入稳定，则在有限时间内 flush pending push
  -> 如果超时，保留 pending 状态，稍后继续
```

## 8. Tracked Paths

同步核心只关注这些路径：

```txt
entries/
media/
annotations/
reviews/
manifest.json
```

应用端传入的 `changedPaths` 必须是仓库相对路径。共享同步核心会过滤危险路径、未知路径和临时路径。这样保存单篇日记时可以避免扫描整个 worktree；未知来源或手动全量同步时，仍可回到 tracked scope 扫描。

## 9. 回流与派生数据

远端更新回到本地后，不直接修改 UI 内存状态，而是让应用在安全时机重新读盘：

- 桌面端：`desktopSyncManager` 在 pull 或 full sync 后，如果编辑器不脏，会重新加载当前日期。
- 移动端：`mobileSyncManager` 在 pull 或 full sync 后，如果保存状态允许，会重新加载今日内容，并通过 `journalEffects.afterRemoteUpdatesApplied()` 刷新 widget snapshot。
- Review：如果当天 review 文件不存在，移动端会按需生成 `reviews/YYYY/MM/YYYY-MM-DD.json`；新建 review 会作为 changed path 进入同步。

## 10. 当前技术事实

- 桌面端：Electron 41、React 19、Vite 7、Tailwind CSS 4。
- 移动端：Expo SDK 56、React Native 0.85、React 19、NativeWind 4。
- 同步：`isomorphic-git` 1.38，HTTPS + GitHub token，GitHub 私有仓库。
- 测试：Vitest 负责 unit / integration，Playwright 负责桌面和真实 GitHub 同步 E2E，Maestro 负责移动端原生 E2E。

## 11. 仍需治理

- 媒体体积：导入图片已做 WebP 压缩和长边限制，仍需要 repo 体积提示和历史媒体治理策略。
- 导入导出：Markdown + media 包格式仍未定。
- 历史日记编辑：移动端当前以今日记录和历史回看为主，历史编辑需要另行设计。
- 冲突 UI：同步核心会保留 true conflict 标记并停止自动 push，但还没有专门的冲突解决界面。
